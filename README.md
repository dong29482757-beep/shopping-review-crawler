## 데이터 장점

### 1. 즉시 활용 가능한 레이블 보유
별점 기반 감성 레이블(`sentiment`)이 이미 부여되어 있어 별도 수작업 레이블링 없이 **감성 분류 모델 학습에 바로 사용** 가능합니다.

### 2. 도메인 특화 (화장품)
화장품 카테고리에 집중된 데이터로, 일반 리뷰 데이터 대비 **도메인 특화 분석에 유리**합니다.  
성분, 제형, 향, 지속력 등 화장품 특유의 표현이 풍부하게 포함되어 있습니다.

### 3. 장기간 데이터 (2015~2026)
10년치 리뷰 데이터를 포함해 **트렌드 분석, 시계열 분석**이 가능합니다.

### 4. 키워드 분류 체계 보유
20개 카테고리 키워드로 분류되어 있어 **카테고리별 비교 분석**이 용이합니다.

### 5. 노이즈 제거 완료
체험단·협찬·도배 리뷰를 제거해 **실구매자 리뷰만** 남아있습니다.

## 쿠팡 차단 우회 방법

쿠팡은 **Akamai Bot Manager** 를 사용해 일반적인 requests나 Selenium으로는 바로 차단됩니다.

### 왜 일반 방식이 안 되는가

Akamai는 5단계로 봇을 탐지합니다.

1. HTTP 헤더, TLS 지문 분석
2. `_abck` 쿠키 기반 세션 검증
3. `navigator.webdriver` 등 브라우저 자동화 흔적 탐지
4. 마우스/키보드/스크롤 패턴 분석
5. IP 평판, 요청 빈도

requests는 1단계에서, Selenium은 3단계에서 차단됩니다.

### 해결 방법: 실제 Chrome + CDP 연결

```
실제 Chrome 실행 (--remote-debugging-port=9222)
       ↓
기존 Chrome 프로필 쿠키를 임시 폴더에 복사
       ↓
Playwright가 connect_over_cdp()로 연결
       ↓
검색창에 키워드를 한 글자씩 타이핑 (URL 직접 접속 X)
```

### 리뷰 데이터 수집 방식

HTML 파싱 대신 **브라우저 내부 API 응답을 가로채는 방식** 사용

```
상품 페이지 접속 → 상품평 탭 클릭
       ↓
브라우저가 내부 API 자동 호출
https://www.coupang.com/next-api/review?productId=...
       ↓
Playwright response interceptor로 JSON 응답 캡처
       ↓
rData → paging → contents 경로로 리뷰 추출
```

2페이지부터는 `page.request.get()`으로 직접 호출하며,  
별점 필터(`ratings=1`, `ratings=2`)를 별도 요청해서 낮은 별점 리뷰도 일정 수 이상 확보합니다.

### 차단 방지 장치

- 상품 사이 랜덤 딜레이 (2~4초)
- 상품 5개마다 쿠팡 홈 방문 + 15~30초 휴식
- 키워드 전환마다 1~2분 대기
- 랜덤 스크롤로 사람처럼 행동 시뮬레이션

## BeautyScope 서비스 (쿠팡+무신사+올리브영 통합)

쿠팡 단독 크롤링 데이터(86,379건)에서 시작해서, 동료가 크롤링한 무신사·
올리브영 데이터까지 더 모아 **538,774건** 규모로 키운 뒤, 그 데이터로
감성 분류 모델(ML+DL)을 학습하고 Streamlit 서비스(BeautyScope)로 구현한
프로젝트입니다. 5분 발표용 슬라이드는 [BeautyScope_발표.pptx](BeautyScope_발표.pptx),
단계별 문제/해결 과정 전체는 [DEVLOG.md](DEVLOG.md)에 정리되어 있습니다.

> `konlpy`(Okt 형태소분석)는 내부적으로 JVM을 쓰므로 Java(JRE 8+)가 설치되어 있어야 합니다.

### 서비스 설계 의도

평점 하나로 합쳐서 보여주면 속성별 차이가 묻힌다. 실제 데이터로 확인한 예:

| 상품 | 평점 | 보습/수분 긍정비율 | 트러블/자극 긍정비율 |
|---|---|---|---|
| 크리스마 더마랩 토너 | 4.55 | 80.5% | 23.3% |
| 닥터지 에이클리어 밸런싱 토너 | 4.78 | 81.5% | **40.5%** |

평점 차이는 0.2점뿐인데, 트러블/자극 긍정비율은 23.3%와 40.5%로 약 2배
차이가 난다. 리뷰어가 중요하게 보는 속성은 사람마다 다르므로(보습/트러블/
향 등), 평점이 아니라 **속성 단위로 데이터를 분해해 보여주는 것**을
서비스의 핵심 구조로 잡았다.

화장품 리뷰는 한 상품에 수백~수천 건씩 쌓여서 사람이 다 읽고 판단하기도
어렵다. 그래서 두 가지를 ML/DL로 해결했다.

1. **감성 분류** — 리뷰 텍스트 → 긍정/중립/부정 (ML: TF-IDF+로지스틱회귀,
   DL: numpy로 직접 구현한 신경망 + 추가로 PyTorch LSTM/KoELECTRA 비교실험)
2. **속성기반 감성분석(ABSA)** — 리뷰를 속성별(보습/향/트러블 등)로 쪼개서
   "이 상품은 보습력은 좋은데 향은 별로다" 같은 세부 정보를 추출하고,
   이를 바탕으로 속성별 랭킹·대안 상품 추천까지 제공해 "평점 높은 제품"이
   아니라 "나에게 맞는 속성을 가진 제품"을 찾아주는 매칭 서비스로 발전시켰다

### 데이터 수집 — 장점과 한계

| 단계 | 데이터 | 건수 |
|---|---|---|
| 1차 | 쿠팡 크롤링 (20개 키워드) | 86,379건 |
| 2차 | + 동료의 무신사 크롤링 데이터 추가 | 230,870건 |
| 2차 | + 동료의 올리브영 크롤링 데이터 추가 (JSONL 4개 파일) | 221,525건 |
| **최종** | 스키마/라벨 통일 + 중복 제거 후 통합 | **538,774건** |

> 1차 쿠팡 크롤링 자체도 순탄하지 않았다 — 쿠팡은 Akamai Bot Manager로
> requests/Selenium을 자동 차단해서, 실제 Chrome을 Playwright로 원격
> 연결하는 방식으로 우회해야 했다. 자세한 내용은 [위 "쿠팡 차단 우회
> 방법"](#쿠팡-차단-우회-방법) 참고.

**장점**: 별점이 곧 라벨이 됨(4~5점 긍정, 3점 중립, 1~2점 부정)이라 별도
수작업 레이블링이 필요 없었고, 화장품 도메인 특화 + 2015~2026년 10년치
데이터로 트렌드 분석도 가능, 20개+ 카테고리 × 8개 속성으로 비교 분석이
용이하다.

**한계**: 별점 기반 라벨링의 전제는 "별점 = 진짜 만족도"인데, 실제로는
별점과 리뷰 내용이 안 맞는 경우가 있다(오기입·과장 등). 독립적인 텍스트
기반 분석과 비교했을 때 별점 5점 리뷰 중 21%가 실제로는 부정적 내용으로
분류됐다 — 즉 라벨 노이즈가 일정 비율 존재한다는 한계를 인지하고
진행했다(완전히 제거하지 않고, 모델 평가 단계에서 별도로 검증).

### 데이터 준비 — 무엇이 문제였고 어떻게 풀었나

| 문제 | 구체적으로 | 해결 방법 |
|---|---|---|
| 스키마가 전부 다름 | 쿠팡/무신사/올리브영 컬럼명이 다 다름 (CSV 2종 + JSONL 4파일) | 플랫폼별 로더 함수를 따로 작성해 공통 스키마로 변환 후 병합 (`merge_datasets.py`) |
| 카테고리가 쿠팡에만 있음 | 쿠팡은 20개 키워드 크롤링이라 세부 카테고리가 자동으로 생기지만, 무신사는 원본에 카테고리 컬럼이 없고 올리브영은 대분류 4종뿐이라 검색 필터에서 두 플랫폼이 거의 안 잡힘 | `product_name` 키워드로 쿠팡과 같은 세부 카테고리를 추정하는 `classify_category()` 추가 → 무신사 1종→11종, 올리브영 4종→14종으로 확장 |
| 긍정 리뷰가 너무 많음 | 전체 리뷰의 87%가 긍정이라, 모델이 아무 생각 없이 "다 긍정"이라고만 찍어도 87% 맞는 셈 | 부정·중립 리뷰는 그대로 두고, 긍정 리뷰만 솎아내서 세 감성의 비율을 맞춤 |
| 문장이 토막 나서 분석됨 | 리뷰를 한글자씩/공백 기준으로만 잘라서 "좋아요"와 "좋아서"를 완전히 다른 단어로 인식 | 형태소 분석기(Okt)로 단어의 "원래 형태"를 찾고, 의미 없는 조사·어미와 불필요한 단어를 제거 (`preprocessing_ko.py`) |
| 같은 리뷰가 중복 수집됨 | 크롤링을 여러 번 나눠서 진행하다 보니 같은 리뷰가 두 번 이상 수집됨 | 상품·작성자·날짜·내용이 모두 같은 리뷰를 중복으로 판단해 제거 (1,085건) |

### 모델 — 4가지를 비교했다

리뷰 텍스트를 긍정/중립/부정으로 분류하는 모델을 단순한 것부터 복잡한
것까지 비교했다.

| 모델 | 역할 / 강점 | accuracy | macro F1 |
|---|---|---|---|
| ML (TF-IDF + 로지스틱회귀) | 가장 가볍고 빠름 — 실서비스 기본 모델 | **0.909**\* | **0.675**\* |
| DL (numpy 직접구현 신경망) | 비선형 패턴까지 학습 — ML보다 정확도 소폭 우위 | **0.920**\* | **0.711**\* |
| LSTM (PyTorch) | 문장 순서를 학습 — 이 데이터에서는 추가 이득 없음 | 0.734 | 0.671 |
| KoELECTRA-small 파인튜닝 | 적은 데이터로도 가장 높은 정확도 | 0.772 | 0.660 |

\* ML/DL은 실제 서비스 운영 분포로 보정한 수치 (학습 시 다운샘플링한 분포와
실제 비율이 달라서 보정함 — 아래 참고)

**결론**: 4개 모델 모두 90% 안팎까지 끌어올렸고, 사전학습 모델(KoELECTRA)이
데이터를 1/7만 쓰고도 가장 높은 정확도를 냈다. LSTM은 순서 정보를 기대했지만
차이가 없었다 — 짧은 리뷰는 어휘 자체가 강한 신호이기 때문으로 보인다.
실제 서비스에는 가볍고 빠른 ML/DL을 우선 적용했다.

DL 구조: TF-IDF(형태소 토큰, 15,000차원) → Dense(128, ReLU) → Dense(64,
ReLU) → Dense(3, Softmax), mini-batch SGD + momentum(0.9), batch 256,
15 epoch. LSTM/KoELECTRA은 별도 Python 3.11 venv에서 PyTorch로 학습.

### 모델 문제 발견 → 해결 → 지금 상태

**① 발견**: 학습 시 다운샘플링한 비율과 실제 운영 비율(positive 86.7%)이
달라서, 실제 분포로 다시 평가하니 모델이 "중립"이라고 예측한 것 중 80%가
틀렸다 (neutral precision 0.20, label shift).

**② 해결**: 모델을 다시 학습하지 않고, `model_utils.py`에서 추론 결과에
"학습 비율 ↔ 실제 비율" 차이를 곱해 확률을 보정하는 방식(prior correction)
으로 빠르게 개선.

**③ 지금 상태**: ML accuracy 0.806→0.909, DL accuracy 0.890→0.920으로
개선 — 재학습 없이도 실사용 가능한 수준을 확보했다.

자세한 진단/원인/해결 과정은 [DEVLOG.md](DEVLOG.md) "모델 문제점과 개선" 참고.

### 모델을 실제로 어떻게 썼나 — 리뷰 신뢰도 검증

모델을 비교 실험으로 끝내지 않고, 보정까지 마친 ML 모델을 서비스 핵심
지표에 직접 연결했다. 별점 기반 라벨링의 한계(별점 5점 리뷰의 21%가
실제로는 부정적 내용)를 그냥 "한계로 인지하고 넘어가는" 게 아니라, 모델로
직접 검증하는 단계를 추가한 것이다.

- `precompute_model_sentiment.py`: 학습된 모델을 리뷰 20건 이상인 상품의
  전체 리뷰에 한 번 배치로 돌려서, 모델이 텍스트만 보고 판단한 감성과
  별점 기반 라벨이 얼마나 일치하는지(`review_reliability.csv`) 상품별로
  집계
- 상품 리포트에 **"리뷰 신뢰도"** 지표로 노출 — 신뢰도가 평균보다 낮은
  상품은 "속성 분석 결과를 더 신중하게 보라"는 경고를 자동으로 보여줌
  (`app.py`, `report_data.get_reliability()`)

자세한 내용은 [DEVLOG.md](DEVLOG.md) "리뷰 신뢰도 검증" 참고.

### 서비스 — 텍스트 데모에서 "상품 리포트"로 전환, 그다음 더 구체화

처음 버전은 "리뷰 텍스트를 입력하면 감성을 알려주는" 데모였는데, 실제
구매를 고민하는 사람에게는 쓸모가 없다는 피드백을 받았다. **상품 리포트**
기능으로 전환: 상품을 검색하면 8개 속성(보습/수분, 향, 트러블/자극,
발림성/흡수력, 지속력, 가격/가성비, 용기/디자인, 효과)별 긍정 비율과
대표 리뷰를 보여준다. 538,774건 전체에 ML 모델(Okt 형태소분석 필요)을
돌리기엔 느려서, 정규식 키워드 매칭 기반의 규칙 기반 속성분석(`absa.py`)
으로 전체 데이터를 52초에 처리했다.

여기서 한 번 더 "조금 더 구체적이고 실용적이었으면 좋겠다"는 피드백을
받고, 데이터 대시보드와 리뷰 텍스트 데모 탭을 빼고 ABSA 데이터를 더
적극적으로 쓰는 기능 두 개를 추가했다 (`report_data.py`):

- **속성별 랭킹**: 카테고리+속성을 고르면 그 속성이 가장 좋은 상품
  Top 10을 보여준다 ("보습력 좋은 토너 Top 10")
- **대안 상품 추천**: 지금 보는 상품의 가장 약한 속성을 찾아서, 같은
  카테고리에서 그 속성이 더 나은 상품을 추천 ("이거 말고 다른 건?").
  절대 기준(긍정비율 80% 이상)으로 후보를 거르면 속성마다 기본 수준이
  달라서(트러블/자극은 언급 자체가 불만 맥락이라 전체 평균이 낮음) 후보가
  거의 안 나오는 문제가 있었음 → "현재 상품보다 충분히 나은가"(+10%p
  이상)로 기준을 바꿔서 해결
- **피부타입별 세그먼트 분석**: 무신사 원본에 있던 `skin_type`(피부타입)
  필드를 스키마 통합 과정에서 버렸었는데, 다시 추출해서 건성/지성/복합성/
  민감성별로 속성 평가를 따로 보여준다 (`precompute_skin_segments.py`,
  147,966건에서 피부타입 식별, 1,202개 상품×피부타입 세그먼트)

예: 어떤 토너는 전체 평점이 4.9인데 트러블/자극 속성만 보면 부정 비율이
67% — 평점만 봐서는 알 수 없는 정보를 드러낸다. 자세한 내용은
[DEVLOG.md](DEVLOG.md) "문제 2", "서비스 고도화" 참고.

세 번의 피드백(실용성 없음 → 더 구체적으로 → 속성 매칭까지)을 거치며
**"평점이 높은 제품"을 보여주는 서비스에서 "내가 중요하게 생각하는 속성에
강한 제품"을 찾아주는 매칭 서비스**로 단계적으로 발전했다.

### 배포 준비

- `requirements.txt`로 의존성 고정, 학습된 모델/집계 데이터를 `models/`에
  포함해 별도 학습 없이 바로 서비스 구동 가능
- 538,774건 원본 대신 사전 집계 CSV(`agg_*.csv`, `product_summary.csv` 등)를
  서비스가 읽도록 분리해 로딩 속도 확보
- 대용량 원본 크롤링 CSV는 `.gitignore` 처리, 정제된 `merged_reviews_all.csv`와
  모델/집계 파일만 추적

### 실행 방법

```bash
# 1. 데이터 통합
python merge_datasets.py

# 2. 학습 데이터 준비 (클래스 밸런싱) + 형태소 토큰화
python ml/prepare_data.py
python ml/tokenize_data.py

# 3. 모델 학습
python ml/train_ml.py
python dl/train_dl.py

# 4. 상품 리포트용 집계 생성 (속성분석 + 피부타입 세그먼트)
python precompute_absa.py
python precompute_skin_segments.py

# 5. 서비스 실행 (fileWatcherType none 권장 — 불필요한 재시작/연결끊김 방지)
streamlit run app.py --server.fileWatcherType none
```
