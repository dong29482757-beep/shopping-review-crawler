# BeautyScope 개발 기록

> 화장품 리뷰 크롤링 데이터로 감성 분류 모델을 만들고, Streamlit으로 서비스까지
> 구현한 과정의 기록. 발표용 정리 문서.

## 1. 서비스 개요

**BeautyScope**: 쿠팡·무신사·올리브영 3개 플랫폼의 화장품 리뷰를 학습해
리뷰 텍스트의 감성(긍정/중립/부정)을 분류하는 Streamlit 웹 서비스.

- 리뷰를 입력하면 ML/DL 두 모델이 동시에 예측 결과를 보여줌
- 538,774건 데이터 기반 통계 대시보드 (플랫폼/카테고리별 분포, 트렌드, 키워드)

## 2. 데이터 준비 단계 (가장 시간이 많이 든 단계)

### 2-1. 1차 수집 — 쿠팡 단독 (86,379건)
- 20개 키워드로 쿠팡 화장품 리뷰 크롤링, 별점 기반 sentiment 라벨 부여
  (4~5점=positive, 3점=neutral, 1~2점=negative)
- **문제**: 체험단/협찬/도배성 리뷰가 섞여 있어 노이즈가 심함
- **해결**: `remove_noise.py`, `quality_check.py`로 패턴 기반 필터링 후
  112,971건 → 86,380건으로 정제

### 2-2. 데이터 추가 확보 — 다른 사람이 크롤링한 데이터 통합
연구 규모를 늘리기 위해 동료가 크롤링한 무신사(231,029건)·올리브영(221,525건,
JSONL 4개 파일) 데이터를 추가로 확보. **이 단계에서 컬럼 스키마가 셋 다 다 달라서
바로 합칠 수 없었음** — 이게 가장 까다로운 부분이었다.

| 문제 | 내용 | 해결 |
|---|---|---|
| 컬럼명 불일치 | 쿠팡 `review_content`/무신사 `review_text`/올리브영 JSON 키 다름 | 공통 스키마(`platform, product_id, rating, sentiment, review_content, review_date` 등)로 통일하는 매핑 스크립트 작성 |
| **sentiment 라벨 기준 불일치 (가장 중요)** | 무신사의 `sentiment` 컬럼은 별점이 아니라 **리뷰 텍스트 분석 기반**이었음. 확인해보니 5점 리뷰의 21%가 `부정`으로 라벨링되어 있었음 (`rating=5` & `sentiment=부정` 42,869건) | 신뢰할 수 없는 라벨이라 판단, **모든 데이터의 sentiment를 별점에서 일괄 재계산**해서 기준을 통일. 원본 라벨은 `sentiment_text_raw`로 보존만 함 |
| 인코딩 문제 | 무신사 CSV가 BOM 포함 UTF-8(`utf-8-sig`)인데 기본 인코딩으로 읽으면 한글이 깨짐(mojibake) | `encoding='utf-8-sig'` 명시 |
| 형식 불일치 | 올리브영은 CSV가 아니라 JSONL 4개 파일(카테고리별), 쿠팡/무신사는 CSV | 파일별 로더 함수를 따로 만들어 통일 스키마로 변환 후 `pd.concat` |
| 중복 리뷰 | 같은 리뷰가 여러 크롤링 시점에 중복 수집됨 | `product_id+nickname+date+content` 기준 `drop_duplicates` (1,085건 제거) |

최종 통합 데이터: **538,774건** (`merge_datasets.py`로 재현 가능)

| 플랫폼 | 건수 |
|---|---|
| 무신사 | 230,870 |
| 올리브영 | 221,525 |
| 쿠팡 | 86,379 |

### 2-3. 클래스 불균형 문제
- 통합 후 positive 86.7%, negative 8.2%, neutral 5.1% — 이대로 학습하면
  모델이 전부 positive로 찍어도 86%대 정확도가 나와서 무의미한 모델이 됨
- **해결**: negative/neutral 전량 + positive는 negative의 2배 규모로만
  다운샘플링 (`ml/prepare_data.py`) → 학습셋 160,176건 (positive 88,572 /
  negative 44,286 / neutral 27,318)

## 3. 모델 — ML과 DL을 나눠서 비교

### 3-1. 환경 제약 (예상치 못한 불편함)
- 원래 PyTorch로 LSTM/BERT 계열 파인튜닝을 하려 했는데, **이 환경의 Python이
  3.14라서 PyTorch/TensorFlow 휠이 아직 배포되지 않아 설치 자체가 실패**
  (`pip install torch` → `No matching distribution found`)
- 다른 파이썬 버전도 환경에 없어서 venv 교체도 불가능
- **해결**: 신경망 프레임워크 없이 **numpy로 순전파/역전파를 직접 구현**해서
  DL 모델을 만듦. 오히려 "신경망이 내부적으로 어떻게 학습되는지"를 발표에서
  더 명확히 설명할 수 있는 소재가 됨

### 3-2. ML 모델 — TF-IDF + LogisticRegression
- 형태소 분석기(konlpy)는 자바 의존성이 있어 설치가 무거워서, 1~2gram 음절
  단위 TF-IDF(`max_features=20000`)로 대체 (별도 설치 없이 비슷한 효과)
- `class_weight='balanced'`로 불균형 보정
- 학습 시간 6.6초 (CPU)
- **결과: accuracy 0.738 / macro F1 0.677**

```
              precision    recall  f1-score   support
    negative       0.75      0.72      0.74      6547
     neutral       0.39      0.53      0.45      4056
    positive       0.89      0.81      0.85     13423
```

### 3-3. DL 모델 — numpy 피드포워드 신경망 (직접 구현)
**신경망 구조**: TF-IDF(20,000차원) → Dense(128, ReLU) → Dense(64, ReLU) →
Dense(3, Softmax)

**학습 방식**:
- Mini-batch SGD + momentum(0.9), L2 정규화
- learning rate 0.08에서 epoch마다 0.92배 감소
- batch size 256, 15 epoch, 학습 시간 약 132초
- loss: cross entropy

**구현 포인트**: TF-IDF가 sparse 행렬이라 numpy 기본으론 처리가 안 돼서
scipy sparse와 dense 배열 간 행렬곱(`X @ W1`, `X.T.dot(delta)`)을 직접
backprop 수식에 맞춰 구현함.

**결과: accuracy 0.762 / macro F1 0.663**

```
              precision    recall  f1-score   support
    negative       0.73      0.76      0.75      6547
     neutral       0.44      0.32      0.37      4056
    positive       0.84      0.89      0.87     13423
```

**관찰한 점**: epoch 4 부근(test_acc 0.781)을 찍고 이후로는 train loss는
계속 떨어지는데 test accuracy는 오히려 진동/하락 — **과적합 패턴**을 직접
확인함. 다음 개선으로 early stopping이나 dropout 추가가 필요하다는 결론.

### 3-4. ML vs DL 비교
| | accuracy | macro F1 | 학습시간 |
|---|---|---|---|
| ML (LogReg) | 0.738 | **0.677** | 6.6s |
| DL (직접 구현 NN) | **0.762** | 0.663 | 132s |

단순 TF-IDF 특징에서는 가벼운 로지스틱 회귀가 매크로 F1 기준으로 더 안정적이고,
신경망은 정확도는 약간 높지만 과적합 경향이 있어 클래스 불균형(neutral)에는
더 취약했음. neutral 클래스가 항상 가장 어려운 클래스였는데, 이는 "중립" 자체가
별점 3점이라는 모호한 기준에서 나온 라벨이라 텍스트 신호가 약한 것이 근본 원인.

## 3-5. 모델 문제점과 개선 — 실제 서비스 분포로 재평가

학습/검증에서 본 accuracy(0.738 / 0.762)는 **클래스 밸런싱을 한 데이터셋** 기준이라,
실제 운영 환경(원본 비율: positive 86.7% / negative 8.2% / neutral 5.1%)에서
그대로 쓰면 성능이 다를 수 있다는 의심이 들어 실제 비율 그대로의 5만 건
샘플로 다시 평가해봤다.

### 발견한 문제
실제 분포로 평가하니 **neutral 클래스의 precision이 0.20까지 떨어짐** —
모델이 "중립"이라고 예측한 것 중 80%가 틀린 것. 원인은 학습할 때 클래스
불균형을 해소하려고 positive를 다운샘플링해서 학습셋 비율(긍정 55% / 부정
28% / 중립 17%)을 만들었는데, 이 비율이 실제 운영 비율과 크게 다르기 때문.
모델은 학습 시 본 비율을 기준으로 확률을 추정하므로, 실제 데이터에 적용하면
중립·부정을 과대 예측하게 된다 (label shift / prior shift 문제).

| | 실제 분포에서 accuracy | 실제 분포에서 macro F1 | neutral precision |
|---|---|---|---|
| ML (보정 전) | 0.806 | 0.626 | 0.20 |
| DL (보정 전) | 0.890 | 0.712 | 0.37 |

### 해결 방법
모델이 출력한 확률을 학습 시 클래스 비율과 실제 클래스 비율의 차이로
보정한다 (prior correction):

```
p_real(y|x) ∝ p_model(y|x) × (실제 클래스 비율 / 학습 클래스 비율)
```

`model_utils.py`에 `PRIOR_CORRECTION` 배열을 추가해 모델 출력 확률에
곱한 뒤 다시 정규화(`correct_proba`)하고, 그 보정된 확률로 최종 라벨을
결정하도록 변경. 재학습 없이 추론 단계에서만 보정.

### 보정 후 결과 (같은 5만 건 실제 분포 샘플)

| | accuracy | macro F1 | neutral precision |
|---|---|---|---|
| ML (보정 후) | **0.908** (+0.102) | **0.685** (+0.059) | **0.37** (+0.17) |
| DL (보정 후) | **0.922** (+0.032) | **0.718** (+0.006) | **0.51** (+0.14) |

재학습 없이 추론 시 확률 보정만으로 실제 서비스 환경 정확도가 크게
개선됨. 다만 neutral precision은 여전히 가장 약한 지점 — 별점 3점 기준
라벨 자체가 텍스트 신호와 약하게 연결되어 있다는 근본적인 한계는
모델 보정으로 완전히 해결되지 않음. 다음 개선 과제로 남김 (예: neutral을
별도 임계값으로 다루거나, 4점도 약positive/3~4점 경계를 재정의).

## 3-6. 추가 실험 — LSTM / Transformer (PyTorch)

"다른 사람들은 LSTM이나 Transformer를 쓰는데 너는 numpy FFNN만 썼다"는
피드백을 받고, 실제로 PyTorch 기반 모델을 환경 문제를 우회해서 시도해봤다.

### 환경 문제 재해결
메인 환경 Python이 3.14라서 PyTorch가 안 설치되는 문제는 여전했지만,
`winget install Python.Python.3.11`로 **Python 3.11을 별도로 설치**하고
`torch_env`라는 별도 venv를 만들어 그 안에 PyTorch(CPU)와 transformers를
설치해서 해결. GPU(NVIDIA, 드라이버 462.96/CUDA 11.2)는 너무 오래돼서
최신 PyTorch의 CUDA 빌드와 호환이 안 돼 CPU로 학습.

### LSTM (단어 시퀀스 기반, 사전학습 임베딩 없음)
- 구조: Embedding(100) → Bidirectional LSTM(128) → Dense(3)
- 학습: 136,150건, 6 epoch, CPU로 epoch당 약 30초 (`dl/train_lstm.py`)
- **결과: accuracy 0.734, macro F1 0.671**
- TF-IDF 기반 ML/FFNN과 성능이 거의 동일 — 사전학습 임베딩 없이 처음부터
  학습한 LSTM은 단어 순서 정보를 살리지 못하고 사실상 TF-IDF 수준의
  정보량만 학습한 것으로 보임 (이 데이터 규모/도메인에서는 순서 정보보다
  어휘 자체가 더 강한 신호였다는 뜻)

### Transformer 파인튜닝 (KoELECTRA-small, 사전학습 모델)
- `monologg/koelectra-small-v3-discriminator`를 HuggingFace에서 받아 파인튜닝
- CPU 학습이라 전체 136k건은 시간이 너무 오래 걸려서(배치16 기준 1 step≈0.2s
  → 전체 1 epoch 추정 45분+) **2만 건 서브샘플 / 2 epoch**로 축소해서 실행
  (`dl/train_transformer.py`, GPU 환경이면 `SAMPLE_TRAIN=None`으로 전체 학습 가능)
- 2 epoch 학습시간 약 14분 (epoch1 418s, epoch2 421s)
- **결과: accuracy 0.772, macro F1 0.660** (4,000건 검증)

### 4개 모델 종합 비교

| 모델 | accuracy | macro F1 | 학습 데이터 | 비고 |
|---|---|---|---|---|
| ML (TF-IDF+LogReg) | 0.738 | **0.677** | 136,150건 | 가장 가벼움, 6.6초 |
| DL FFNN (numpy 직접구현) | 0.762 | 0.663 | 136,150건 | 프레임워크 없이 구현 |
| LSTM (PyTorch, 사전학습 임베딩 X) | 0.734 | 0.671 | 136,150건 | 순서 정보 활용 기대했으나 차이 없음 |
| KoELECTRA-small 파인튜닝 | **0.772** | 0.660 | 20,000건(서브샘플) | 가장 적은 데이터로도 accuracy 최고치, 단 충분히 학습 안됨(2epoch) |

### 결론
- 이 과제(짧은 리뷰 텍스트, 키워드성 표현이 강한 도메인)에서는 모델
  구조(LSTM/Transformer)보다 **어휘 자체(TF-IDF)가 이미 강한 신호**라
  단순 모델과 복잡한 모델의 성능 차이가 생각보다 크지 않았다
- KoELECTRA는 데이터를 1/7만 쓰고도 다른 모델과 비슷하거나 더 나은
  accuracy를 보여 **사전학습의 효과**는 분명히 확인됨 — 전체 데이터로,
  더 많은 epoch으로, GPU 환경에서 학습하면 격차가 더 벌어질 것으로 예상
  (지금은 CPU 학습 시간 제약으로 검증을 축소함)
- Streamlit 데모(`app.py`)는 메인 환경(Python 3.14)에서 동작해야 해서
  PyTorch 모델(LSTM/KoELECTRA)은 별도 venv(`torch_env`, Python 3.11)에서만
  실행 가능 — 현재 데모는 ML+FFNN 두 모델로 운영, LSTM/Transformer는
  비교실험 결과로만 기록 (다음 단계: 두 환경을 API로 분리하거나 메인
  환경의 Python 버전을 낮춰서 통합)

## 3-7. 피드백 반영 — "서비스가 실용성 없다 / 불용어 처리 안 했다 / 모델이 좋은지 모르겠다"

세 가지 지적을 받고 각각 원인을 진단하고 다시 작업했다.

### 문제 1: 불용어/형태소 처리를 제대로 안 함
이전까지 `re.findall(r"[가-힣]{2,}")`로 음절만 잘라내는 정규식 토크나이저를
썼다. 조사가 토큰에 그대로 섞여서("좋아요"/"좋아서"/"좋은데"가 전부 다른
토큰) TF-IDF가 같은 의미의 단어를 다른 특징으로 취급했고, 부정/긍정 키워드
집계도 의미 없는 조사·접사가 섞여 나왔다.

**해결**: `preprocessing_ko.py`에 Okt(코엔엘파이) 형태소 분석을 추가해
명사/동사/형용사/부사만 남기고 어간으로 정규화(stem=True)했다. 추가로
직접 만든 불용어 리스트(`STOPWORDS`)로 "제품", "사용", "구매" 같은 주제와
무관한 범용 단어를 제거했다. Okt가 화장품 도메인 복합명사를 잘못 쪼개는
문제(예: "재구매"→"재"+"구매", "발림성"→"발림"+"성은")는 분석 전에
플레이스홀더로 치환해서 분리를 막는 방식으로 우회했다.

**검증 결과 (중요, 정직하게 기록)**: 형태소 분석을 적용해도 ML/DL 모델의
accuracy·macro F1은 거의 변화가 없었다 (오히려 근소하게 하락하는 경우도
있었음). 원인으로 추정되는 것: ① 짧은 화장품 리뷰는 음절 단위로도 이미
충분한 신호가 담겨 있었고, ② Okt의 불완전한 형태소 분리가 일부 정보
손실을 일으켰다. 그래도 불용어/조사를 제대로 처리한 것 자체는 키워드
집계(대시보드)의 가독성을 크게 개선했다 — 이전엔 "좋다"가 "좋아요/좋아서/
좋은데"로 쪼개져서 빈도가 분산됐는데, 지금은 정규화된 어간으로 합쳐져서
실제 다빈도 단어가 제대로 보인다.

### 문제 2: 서비스에 실용성이 없음 (텍스트박스에 리뷰 쳐보는 데모)
지적이 정확했다. "리뷰 텍스트를 입력하면 감성을 알려준다"는 기능은 실제
화장품을 살지 고민하는 사람에게 쓸모가 없다 — 이미 써본 리뷰의 감성을
사람이 직접 텍스트로 타이핑해서 넣을 이유가 없기 때문.

**해결**: **상품 리포트** 기능을 새로 만들었다. 상품을 검색하면:
- 보습/수분, 향, 트러블/자극, 발림성/흡수력, 지속력, 가격/가성비, 용기/디자인,
  효과 — 8개 속성별로 긍정 비율을 보여준다 (속성기반 감성분석, ABSA)
- 대표 긍정/부정 리뷰를 바로 보여준다

이건 실제로 "이 토너 평점은 4.9인데, 자극/트러블 항목만 보면 부정 비율이
67%다" 같은, **전체 평점만 봐서는 알 수 없는 정보**를 제공한다. 538,774건
전체에 ML 모델을 돌리려면 Okt 형태소분석이 병목(초당 약 200건)이라 비용이
너무 커서, 정규식 키워드 매칭 + 부정어 처리 기반의 규칙 기반 ABSA(`absa.py`)로
구현해 전체 데이터를 52초에 처리했다. 정확도는 ML 분류기보다 거칠지만
속도와 커버리지 트레이드오프로 선택.

### 문제 3: 모델이 좋은지 모르겠음
"좋다"를 어떻게 판단하는지가 불명확했던 게 문제였다. 단순 accuracy 숫자
하나만 보여주는 건 모델 품질을 보여주는 좋은 방법이 아니다. 그래서:
- 클래스 밸런싱한 학습셋 성능과 **실제 운영 분포로 보정한 성능**을 둘 다
  기록 (`model_utils.py`의 `PRIOR_CORRECTION`, 위 3-5절)
- LSTM/Transformer 등 다른 구조와 직접 비교 실험 (위 3-6절)
- 속성별 분해(ABSA)로 "전체 정확도"가 가리는 약점(neutral 클래스, 특정
  속성에서의 오류)을 드러냄

## 4. 서비스 구현 (Streamlit)

- `app.py`: 4개 탭 구성
  1. **상품 리포트** (핵심 기능) — 상품 검색 → 속성별(보습/향/트러블/발림성/
     지속력/가격/용기/효과) 긍정 비율 + 대표 긍정/부정 리뷰
  2. 리뷰 텍스트 분석 — 텍스트 입력 → ML/DL 동시 예측 + 확률 시각화
  3. 데이터 대시보드 — 플랫폼별 분포, 별점 분포, 월별 트렌드, 카테고리별
     감성 비율, 부정/긍정 다빈도 키워드(형태소 분석 기반), 리뷰 샘플 탐색
  4. 프로젝트 소개
- **문제**: 538,774건 원본 CSV를 대시보드 로딩 시마다 읽으면 느림
- **해결**: `precompute_dashboard.py`로 집계 결과를 미리 작은 CSV로 만들어두고
  대시보드는 그 집계 파일만 읽도록 분리 (`@st.cache_data`로 추가 캐싱)
- 로컬에서 `streamlit run app.py` 구동 후 HTTP 200 확인, 텍스트 입력 →
  ML/DL 예측 정상 동작 확인

## 5. 배포 준비
